Domain Driven UX

Domain Driven Design is wildly used approach to software development. You could consider this library as an attempt to use the same principles to define user interfaces inside frontend data driven applications. Domain expert could describe customer invoices list in following terms

We define here data binding and in which format displayed value should be generated, the actual UX rendering is the concern of the holistic library.

    Few subtle points worth to mention here
  • 'kind' property mark column as containing cutom component inside, you must configure these on app start.
  • There is implicit value binding here between row data and component value.
  • To load data inside the table use DataProvider interface.

Why use custom component for seemingly string values ?

When customer name is displayed inside table cell it could be just a string, if so we don't need some custom component to render it
    but what if customer name
  • Has link to the customer profile to open
  • Has some popup to show with additional info
  • Format name value, such as cappitalise first letters (it could be achived also via formatters)
If you create some custom component for this case and configure it to be used in table definitions then you could use it in any table inside application by simple reference it inside table definition.
